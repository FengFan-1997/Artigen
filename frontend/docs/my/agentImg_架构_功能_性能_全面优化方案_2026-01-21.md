### agentImg 架构 / 功能 / 性能 全面优化方案（规划稿）

本规划基于仓库既有工程规范与文档体系（路由、状态管理、API 层、缓存、埋点、国际化、登录付费等）以及 agentImg 当前实现现状，目标是把 agentImg 作为一个可持续迭代的“可组合工作流产品”，在不破坏既有全站架构的前提下，系统性提升：可维护性、可扩展性、性能与成本、稳定性、可观测性、商业转化。

---

## 1. 背景与目标

### 1.1 agentImg 的定位（建议统一定义）

- 产品定位：面向电商图/商业视觉的 AI 生成与编辑工作流（从“描述/参考图/品牌资产”到“出图/下载/复用/历史沉淀”）
- 形态定位：站内一个相对独立的业务域（`src/agentImg`），但共享全站能力（登录、点数、埋点、i18n、路由、构建与拆包策略）

### 1.2 北极星目标（建议可量化）

- 生成成功率：`img2img success / img2img request` ≥ 97%
- 首次可用时间（TTI）：P75 ≤ 3.0s（移动端）
- 交互响应：输入/切换侧栏/打开历史等关键交互 P95 ≤ 100ms
- 单次生成端到端耗时：P50 ≤ 25s，P95 ≤ 70s（区分 deep/quick、是否带参考图）
- 商业指标：新用户完成首次生成率、付费转化率、复购（点数）等

## 2. 现状梳理（代码结构与关键数据流）

### 2.1 模块边界（当前）

- 页面入口：`src/agentImg/index.vue`
  - UI/交互（聊天视图 + 产品侧栏 + 历史侧栏 + 下载弹窗）
  - 通过 composables 组合业务能力：auth、credits、ui、models、upload、settings、flow、history、generation
- Composables（核心逻辑分层雏形已存在）：
  - 认证：`src/agentImg/composables/useAgentImgAuth.ts`
  - 点数：`src/agentImg/composables/useAgentImgCredits.ts`
  - 生成编排：`src/agentImg/composables/useAgentImgGeneration.ts`
  - 深度思考（方向分析 + prompt 生成）：`src/agentImg/composables/useAgentImgFlow.ts`
  - 历史：`src/agentImg/composables/useAgentImgHistory.ts`
  - 上传与预览：`src/agentImg/composables/useAgentImgUpload.ts`
  - 设置（产品档案）：`src/agentImg/composables/useAgentImgSettings.ts` + `src/agentImg/stores/settings.ts`
  - 文案：`src/agentImg/composables/useAgentImgLocale.ts`
- 服务层：`src/agentImg/services/text.ts`
  - `generateText()`：调用 `/api/generate`（固定文本模型）
  - `img2img()`：调用 `/api/img2img`（固定图像模型）
  - 自带 requestId、timeout、AbortSignal、输入图片归一化与安全过滤

### 2.2 核心数据流（建议作为后续架构基线）

#### A. Quick Mode（快速生成）

1. 用户输入（`flow.userInput`）
2. `generation.doPrimary()`：
   - ensureAuthed → 埋点 → 清取消态 → abort/cancel 旧请求
   - 组装 prompt（产品档案 + 用户需求）
   - 组装 negativePrompt（安全负向 + 可选 logo 放行策略）
   - 读取参考图与生成缩略图（用于历史展示）
   - 写入 `history` 一条 item（先 image=null）
   - 调用 `img2img()` → 成功后回填该 item 的 image
   - 刷新点数余额

#### B. Deep Mode（深度思考）

1. 第一次点击生成：
   - `flow.analyzeDirections()` → `generateText()` → 解析 JSON → 得到 4 个方向 options
2. 用户选择方向后再次点击生成：
   - `useAgentImgGeneration` 拼装深度 prompt（用户需求 + 方向标题/摘要/styleTags）
   - 拼装 negativePrompt（安全负向 + 方向 negativeTags）
   - 清空 options（退出“选择方向态”）→ 进入 img2img
3. 成功后更新历史与点数

## 3. 痛点清单（按影响面归类）

### 3.1 架构与可维护性

- index.vue 仍包含较多“编排胶水逻辑”（prefill 引用、跨 composable 状态同步、交互细节、监听器），随着功能扩展会继续膨胀
- `GenerationDeps` 用 `any` 较多，依赖注入边界不够清晰，后续重构成本上升
- 同类逻辑存在两套做法：
  - 全站 API 层主要基于 axios request 拦截（见文档 API 层/类型工具库），agentImg 目前在 `services/text.ts` 内使用 fetch
  - 国际化体系全站是“vue-i18n + 自定义 $t”双系统，agentImg 目前走 `useLanguageStore` + 自己的 ui 字典，后续跨页复用文案会变难

### 3.2 体验与功能扩展

- deepMode 与上传参考图存在互斥（“图生图暂不支持深度思考”），但从产品角度深度思考对图生图也很关键（构图/光影/材质补全）
- 生成结果只有单张（`img2img()` 取第一张），而 settings store 已有 `imageCount` 概念但未落地
- 历史能力：
  - guest 用 localStorage，登录用服务器；缺少“统一历史视图 + 合并策略 + 搜索/过滤/标记/复用”
  - 缺少“Prompt 版本化”与可重现参数（seed/steps/size 等）的完整记录与复用入口
- 下载逻辑是前端 a 标签直链下载（且 2K/4K 仅 UI），缺少后端对不同分辨率与水印策略的真实支持对齐

### 3.3 性能与成本

- 上传预览：
  - 预览与缩略图生成在主线程（Canvas/图片解码），移动端和低端机可能卡顿
  - 大图压缩策略存在但偏粗粒度（>2.5MB 才压缩、maxDim=1536、jpeg 0.92），缺少更精细的“按目标用途”策略（预览/参考/缩略图/上传）
- 日志与隐私：
  - `services/text.ts` 会 `console.log` 输出 prompt（可能包含用户敏感信息），也会输出 imagesCount 等；生产环境风险较大

### 3.4 稳定性与可观测性

- 错误码体系与全站 request.ts 体系未统一，后续“登录/付费拦截、450 兜底弹窗、全局错误上报”等能力难以复用
- 请求生命周期可观测性不足：
  - 有 requestId，但没有贯穿“埋点/错误/成功/耗时/取消”标准字段
  - deepMode 两段式（directions→final/img2img）缺少统一 trace

## 4. 总体优化策略（分层架构 + 工作流产品化）

### 4.1 建议目标架构（agentImg 内部）

保持 agentImg 作为独立业务域，但内部结构更清晰地分层：

- **UI 层（components/views/index.vue）**：只做展示与事件派发，不做复杂业务编排
- **Hooks/Workflow 层（优先对齐 Hooks 体系）**：定义“用户任务”的状态机与编排，建议以 Hook 的形式组织（例如放在 `src/agentImg/workflows/*`，或抽到 `src/hooks/agentImg/*` 便于跨业务复用）
- **Domain 层（domain/*）**：纯函数与领域模型（PromptSpec、HistoryItem、Credits、ModelPolicy、DownloadPolicy 等）
- **API/Service 层（对齐 src/api）**：对接后端 API（generate、img2img、history、download、pricing），建议最终收敛到 `src/api` 模块化结构，并统一走 `@/utils/request`
- **Infra/Utils 层（对齐 src/utils）**：缓存、序列化、错误码、日志、埋点封装、图片处理（可引入 worker），尽量复用全站 `src/utils` 的既有能力

该架构与全站“UI → Hooks/Utils → API → Store”一致（详见系统集成与数据流文档），只是把 agentImg 的“Hooks”进一步产品化为“Workflows”。

### 4.2 标准化“生成任务”状态机

把目前散落在 `useAgentImgFlow`、`useAgentImgGeneration`、`index.vue` 的状态，收敛成统一状态机，核心字段建议如下：

- `taskId`（= requestId / traceId）
- `mode`：quick | deep_directions | deep_generate
- `phase`：idle | preparing | requesting | success | failed | cancelled
- `inputs`：userText、context、refImages、logo、model
- `outputs`：prompt、negativePrompt、params、images[]
- `billing`：costEstimate、costConfirmed、walletBefore/After
- `timings`：prepareMs、requestMs、totalMs

收益：

- UI 只需渲染状态，不参与编排
- 取消/重试/恢复都以状态机为中心
- 埋点与监控可自动从状态机采集

## 5. 架构优化方案（细化到可落地的改造点）

### 5.1 统一请求与错误码体系（与全站对齐）

目标：让 agentImg 的 API 调用能复用全站的“登录/付费拦截、错误码映射、上报链路、取消策略”。

建议做法（两种路径二选一，建议优先 A）：

- A. 将 `services/text.ts` 抽象成“同语义的请求封装”，内部仍用 fetch，但输出统一错误模型（errorCode、httpStatus、needLogin、needPay、retryable、traceId）
- B. 迁移到全站 `@/utils/request` + `src/api` 模块化结构（对齐 `01_API层架构.md` 的约束：所有 API 基于 request.ts），并在 `src/api` 下新增 agentImg 模块承接 `/api/img2img`、`/api/generate`、历史与后处理等接口

配套要求：

- 统一错误码映射：将 `UPSTREAM_TIMEOUT/FETCH_ERROR/OFFLINE/INSUFFICIENT_CREDITS` 等归一到全站可识别的 error 模型
- 统一“需要登录/需要付费”触发机制：当后端返回特定 code/450 时，交给全局弹窗体系兜底（参考文档 登录与付费体系、request.ts）

### 5.2 Prompt 与参数体系版本化（可复现与可迁移）

新增概念：`PromptSpec`（领域模型）

- 字段：`version`、`userText`、`contextText`、`directionOption`、`baseStyleTags`、`safeNegativeTags`、`finalPrompt`、`negativePrompt`、`params`、`model`、`seed`
- 所有历史记录只存 `PromptSpec`（以及输入/输出图片引用），而不是存拼好的字符串

收益：

- 支持“重跑同一规格”（可复现）
- 支持“升级 prompt 生成策略”（做 schema 迁移）
- 支持 A/B（同一 PromptSpec 不同模型/不同参数）

### 5.3 生成编排解耦：把 GenerationDeps 从 any 变成强类型契约

把 `useAgentImgGeneration` 的 deps 切为更小颗粒的接口（并且真实类型化），例如：

- `AuthPort`（ensureAuthed、getUser）
- `CreditsPort`（getWallet、refreshWallet、estimateCost、confirmCost）
- `HistoryPort`（append、update、persist、load）
- `UploadPort`（getRefs、getLogo、createThumbs）
- `PromptPort`（buildQuickSpec、buildDeepSpec）
- `ImageGenPort`（img2img、txt2img、upscale、download）

收益：

- 单测更容易（每个 port 可 mock）
- 新功能（比如多图、批量、不同模型）不会把 deps 越堆越大

## 6. 功能优化方案（产品维度 Roadmap）

### 6.1 工作流产品化：从“聊天”到“任务卡片”

当前交互是 Chat 列表，但本质上用户在做“任务”。建议引入“任务卡片”概念：

- 每次生成形成一张卡：
  - 输入：用户需求 + 产品档案快照 + 参考图/Logo
  - 输出：多张结果图 + 参数（size/steps/seed）+ 模型
  - 操作：继续迭代（续写需求）、变体、重跑、局部编辑、上采样、导出/下载、收藏/对比
- Chat 仍可保留，但作为任务的“对话注释”而不是唯一形态

### 6.2 Deep Thinking 覆盖图生图（解除互斥）

现状 deepMode 与参考图互斥，建议改为：

- deepThink 可用于“图生图增强”：先基于参考图/产品档案/用户需求做方向分析与 prompt 补全，再进入 img2img
- 当存在参考图时，方向输出增加“对齐参考图的要点”：主体材质、镜头语言、光影、背景一致性规则

### 6.3 多图输出与多变体（落地 imageCount）

实现 `imageCount`：

- 让 `img2img` 支持 `n` 张输出（若后端已支持 images 数量参数则直接透传；若不支持，前端并发调用 n 次并使用不同 seed）
- UI：在卡片中展示 2/4/8 变体网格；支持“一键挑选最佳”或“对比视图”
- 成本：将费用估算与扣费确认与 `n` 挂钩（并在 UI 明示）

### 6.4 结果后处理链（与商业交付对齐）

按“电商出图”常见需求提供后处理链路：

- 上采样（2K/4K 真正可用）：前端只展示入口，实际通过后端任务生成并回写 history
- 自动抠图/换背景：与现有 PhotoShoot/AI Background 思路一致，可变成一个统一的“背景工作流”
- 画面清洁（去水印/去文字）策略：与 safeNegative、Logo 放行策略一致
- 导出规格：平台适配（Amazon、Shopify、独立站 hero banner 等），可结合 formatFactory 的现有能力

### 6.5 历史与资产管理

- 登录前后历史合并：
  - guest 历史在登录后可提示“是否迁移到账号”（可选）
- 搜索/过滤：
  - 按产品、风格、颜色、时间、是否含 logo、是否用参考图
- 收藏/标签：
  - 支持把一张结果标记为“模板”，下次一键复用 PromptSpec + 参数

## 7. 性能优化方案（前端、网络、算力成本）

### 7.1 图片处理：主线程减负

目标：让“选图/缩略图/压缩”不阻塞输入与滚动。

- 将 `fileToThumbDataUrl` 与压缩逻辑迁移到 Worker（或 OffscreenCanvas 可用时优先）
- 缩略图策略：
  - 统一 thumb 尺寸（例如长边 360）与质量（jpeg 0.75–0.82），并强制移除元数据
- 上传策略（按用途分级）：
  - 参考图：长边 1024–1536，质量 0.85–0.9
  - Logo：保持清晰但限制分辨率/文件大小，并优先 png
  - 仅预览：长边 512

### 7.2 请求与并发控制

- 生成并发策略：
  - 单任务：禁止并发（已有 abort/cancel）
  - 多变体：采用“受控并发”（例如 2–3 并发），并在 UI 显示进度
- 重试策略：
  - 仅对 retryable 错误（502/503/504/UPSTREAM_TIMEOUT）做指数退避重试（次数与上限可配）
  - 对用户可操作错误（参数不合法/积分不足）不重试

### 7.3 bundle 与首屏

- agentImg 作为独立业务域，可用懒加载进一步隔离：
  - 图片工坊/法律页/市场页等按路由拆分
  - 重组件（例如背景素材、复杂编辑器）采用动态 import
- 与全站构建拆包策略对齐（已有 manualChunks），但 agentImg 内避免引入额外重依赖

## 8. 稳定性与可观测性（埋点 + 监控 + 调试）

### 8.1 标准化埋点（建议最小闭环）

基于全站 `trackEvent` 与 dataLayer 体系（见埋点文档），建议补齐以下事件并统一字段：

- `agentimg_generate_start`
- `agentimg_generate_success`
- `agentimg_generate_fail`
- `agentimg_generate_cancel`
- `agentimg_deep_directions_success/fail`
- `agentimg_download_click`（含分辨率、是否 pro、是否后处理完成）
- `agentimg_model_select`（含是否 locked、是否跳转 market）

统一字段建议：

- `requestId/taskId`、`mode`、`hasRef`、`hasLogo`、`model`、`cost`、`elapsedMs`、`errorCode`

### 8.2 端到端 trace

- directions → img2img 作为同一任务的子 span，前端用 `taskId` 贯穿
- 服务端回写时将 `taskId` 也写入 history item，便于问题定位与复盘

## 11. 分期落地计划（可执行）

### Phase 0（1–3 天）：安全与可观测性底座

- 移除/收敛生产环境 console 输出（prompt、请求体）
- 统一 requestId/taskId 字段，并在埋点事件里携带
- 补齐最小闭环埋点（start/success/fail/cancel + download + model_select）

### Phase 1（1–2 周）：架构收敛与体验提升

- 形成统一的“生成任务状态机”，UI 从状态机读取
- 历史结构升级为 PromptSpec（先兼容旧数据，逐步迁移）
- 引入多变体（2/4）与 imageCount 落地（受控并发 + 成本展示）

### Phase 2（2–4 周）：深度思考覆盖图生图 + 后处理链

- deepThink 支持带参考图的方向分析与 prompt 补全
- 上采样/下载链路改为后端任务（前端展示任务态 + 结果回写 history）
- 资产管理（收藏/标签/搜索）第一版

### Phase 3（持续）：性能与成本优化

- 图片处理 worker 化
- 请求重试与降级策略完善
- 运营配置/模型策略配置化（必要时从 `config.ts` 或后端下发）

## 13. 目标架构蓝图（更细、可直接开工）

本节把“分层架构 + 工作流产品化”落到更具体的模块契约、目录建议、状态机转移、以及与后端/全站能力的对齐方式，确保可以直接按 PR 拆分推进。

### 13.1 建议目录与职责（agentImg 内部）

目标：让任何人打开目录能直观看懂边界，避免逻辑继续堆到 `index.vue` 或某个超级 composable。

建议结构（不要求一次到位，可按 Phase 渐进迁移）：

### 13.2 “生成任务状态机”更详细规格（事件、转移、数据结构）

建议把当前散落状态，统一成一个 `Task` 概念；History 是 Task 的持久化列表。

#### 13.2.1 Task 数据结构（建议字段）

- `taskId`: string（全链路 id，前端生成，贯穿 directions/img2img/download 等）
- `createdAt`: number
- `mode`: `'quick' | 'deep'`
- `phase`:
  - `'idle' | 'preparing' | 'analyzing' | 'generating' | 'postprocessing' | 'success' | 'failed' | 'cancelled'`
- `input`:
  - `userText`: string
  - `contextSnapshot`: string（产品档案快照，避免后续修改影响历史复现）
  - `refs`: `{ kind: 'file' | 'url'; thumbUrl?: string; source?: string }[]`
  - `logo`: `{ thumbUrl?: string; fileName?: string } | null`
  - `selectedModelId`: string
- `deep`（仅 deep 模式）：
  - `directions`: `{ id; title; summary; styleTags; negativeTags; suggested? }[]`
  - `chosenDirectionId`: string
- `promptSpec`：
  - `version`: string
  - `prompt`: string
  - `negativePrompt`: string
  - `params`: `{ imageSize?; steps?; guidanceScale?; seed? }`
- `output`：
  - `images`: `{ url: string; seed?: number; width?: number; height?: number }[]`
  - `bestImageUrl?: string`
- `billing`：
  - `costEstimate`: number
  - `costConfirmed`: boolean
  - `walletBefore?: { available; frozen }`
  - `walletAfter?: { available; frozen }`
- `telemetry`：
  - `startedAt`: number
  - `endedAt`: number
  - `prepareMs`: number
  - `analyzeMs`: number
  - `generateMs`: number
  - `errorCode?: string`
  - `cancelReason?: 'user' | 'timeout' | 'replaced'`

#### 13.2.2 事件与转移（可直接实现）

事件集合：

- `USER_SUBMIT`（用户点击生成）
- `USER_CANCEL`（用户点击停止）
- `UPLOAD_REFS_CHANGED`（参考图变化）
- `DEEP_ANALYZE_SUCCESS/FAIL`
- `DEEP_CHOOSE_DIRECTION`
- `GENERATE_SUCCESS/FAIL`
- `POSTPROCESS_REQUEST/SUCCESS/FAIL`
- `WALLET_REFRESHED`

转移简表（核心路径）：

- quick：
  - idle → preparing → generating → success/failed/cancelled
- deep：
  - idle → preparing → analyzing →（展示 directions）→ generating → success/failed/cancelled
- 取消原则：
  - 任何请求中都允许 `USER_CANCEL`，最终落到 cancelled，并将取消原因写入 task
- 替换原则（用户再次点击生成）：
  - 若有 active task 且 phase 属于 analyzing/generating/postprocessing，则先 cancel old task（cancelReason=replaced）再启动新 task

### 13.3 PromptSpec 版本化与兼容策略（更具体）

建议引入版本号，避免后续 prompt 结构升级导致历史不可复现。

- `v1`（现状兼容）：存字符串 prompt/negativePrompt 与可选 params
- `v2`（建议）：增加来源字段，便于复现与调试：
  - `source.userText`
  - `source.contextSnapshot`
  - `source.deepDirection`（若 deep）
  - `policies.safeNegativeVersion`
  - `policies.baseStyleVersion`
- 迁移策略：
  - 历史读取时：若缺 `version` 则视作 v1
  - 写入时：统一写 v2，但保留 v1 的 prompt 以便快速渲染

### 13.4 后端契约建议（为多图/上采样/下载/历史打底）

这里不要求立即改后端，但先把“需要的协议”定义清楚，前后端对齐会极大减少返工。

#### 13.4.1 img2img 支持多图与 seed 管理

建议请求扩展字段：

- `n?: number`（期望输出张数）
- `seeds?: number[]`（前端可控种子，便于复现与变体）
- `taskId?: string`（与 requestId 并存也可）

返回建议：

- `images: { url: string; seed?: number; width?: number; height?: number }[]`
- `timings?: { queueMs?; modelMs?; totalMs? }`

#### 13.4.2 下载/上采样改为“任务式”

原因：前端 a 标签直链下载无法保证：

- 2K/4K 真实生成
- Pro/水印策略
- 失败重试与进度
- 与历史一致的可追溯

建议接口形态：

- `POST /api/images/postprocess`
  - body: `{ taskId, imageUrl, op: 'upscale'|'watermark'|'export', params }`
  - return: `{ jobId }`
- `GET /api/images/postprocess/:jobId`
  - return: `{ status: 'running'|'success'|'failed', resultUrl?, errorCode? }`

前端把该 job 作为 task 的 `phase=postprocessing` 子过程，最终把 resultUrl 写入 history。

#### 13.4.3 历史接口建议与分页

现有 `GET /api/images/history/:userId?limit=200` 可进一步增强：

- 支持 cursor 或时间分页：`?cursor=...&limit=...`
- 支持过滤：`?hasLogo=1&hasRef=1&product=...`
- 返回结构统一为 Task/PromptSpec，前端无需再做大量猜测解析

## 14. 性能与稳定性：可执行清单（含验收方式）

本节把“方向”拆成工程执行项，并给出验收/量化方式，避免停留在口号。

### 14.1 关键性能预算（建议先落一版）

- 首屏：
  - JS 初始执行 + hydration（如有）≤ 1200ms（P75 移动端）
  - 首次可交互 ≤ 3000ms（P75）
- 交互：
  - 输入框输入不掉帧（连续输入期间 main thread 长任务 < 50ms）
  - 打开/关闭侧栏 P95 < 100ms
- 图片处理：
  - 选图后 300ms 内显示缩略图（P75）
  - 生成历史缩略图不阻塞滚动（滚动期间无 >100ms 长任务）

### 14.2 图片处理 Worker 化（分步落地）

步骤建议：

1. 先把“缩略图生成”迁移到 Worker（收益最大，风险相对低）
2. 再把“参考图压缩”迁移到 Worker
3. 最后做“多用途策略”与“设备能力自适应”

验收方式：

- DevTools Performance：选 3 张 8MB+ 图片，主线程长任务明显减少
- 低端机（CPU 4x slowdown）仍能保持输入流畅

### 14.3 请求稳定性：超时、重试、幂等、取消

#### 14.3.1 幂等（避免重复扣费/重复生成）

- 客户端生成 `taskId`，服务端以 `taskId` 做幂等键：
  - 同 taskId 的重复请求返回同结果或明确错误（DUPLICATE_REQUEST）
- 扣费采用“预扣/确认”或“服务端一次性原子扣费”策略（与点数体系对齐）

#### 14.3.2 重试策略（只对 retryable）

建议统一判断 retryable 的规则（示例）：

- retryable：`API_ERROR_502/503/504`、`UPSTREAM_TIMEOUT`、`FETCH_ERROR`（视情况）
- not retryable：`INSUFFICIENT_CREDITS`、`EMPTY_PROMPT`、`MODEL_NOT_ALLOWED`、`SILICONFLOW_IMAGE_400` 等

默认重试策略：

- 1 次快速重试（1–2s 随机抖动）
- 1 次退避重试（3–6s）
- 总耗时上限不超过 180s（与现有 timeoutMs 对齐）

#### 14.3.3 取消一致性

- 用户取消应保证：
  - UI 明确显示已取消（History item 有 cancel notice）
  - 服务端不扣费或自动回滚（如已扣）
  - 取消后不会“突然回写成功结果”污染当前任务（需要 taskId/phase 校验）

### 14.4 隐私与敏感日志治理（强制项）

建议做法：

- 建立 `agentImgLogger`：
  - 默认不输出 prompt、base64、用户输入
  - 仅在 dev 或明确开关下输出（且自动截断/脱敏）
- 对外链图片引用：
  - 前端只允许用户手动确认引用
  - 后端 proxy 必须做域名白名单、DNS/重定向限制、大小限制、content-type 校验

验收方式：

- 生产构建（或生产环境）控制台不出现 prompt、base64、用户输入

## 16. 模块契约补齐（可直接对照实现）

本节补齐 13 章的“契约”层细节，目的不是一次性重构完，而是先把边界与数据形态定住，后续 PR 可以渐进迁移而不反复返工。

### 16.1 Ports（依赖注入）建议接口形态

把 `useAgentImgGeneration` 中的 `GenerationDeps` 拆为 Ports，每个 Port 一类职责，尽量做到“可以在单测里纯 mock”。以下为建议最小接口（字段名可按现有实现调整，但语义建议保持）：

```ts
export type TaskId = string;

export type AgentImgError = {
  code: string;
  message?: string;
  httpStatus?: number;
  retryable?: boolean;
  needLogin?: boolean;
  needPay?: boolean;
  traceId?: string;
  raw?: unknown;
};

export type ImgInputRef =
  | { kind: 'file'; file: File; thumbDataUrl?: string }
  | { kind: 'url'; url: string; thumbDataUrl?: string };

export type ImgOutput = { url: string; seed?: number; width?: number; height?: number };

export type PromptSpecV2 = {
  version: 'v2';

  policies: { safeNegativeVersion: string; baseStyleVersion: string };

export type Task = {

  mode: 'quick' | 'deep';
```

### 16.2 API 输出的“统一错误模型”规范

无论 `services/text.ts` 继续用 fetch 还是迁移到 axios，都建议对外只输出统一错误模型，避免 UI 到处写 if/else 猜错误类型：

- 必备字段：`code`、`httpStatus?`、`retryable?`、`needLogin?`、`needPay?`、`traceId?`
- code 命名建议：
  - 网络类：`OFFLINE`、`FETCH_ERROR`、`UPSTREAM_TIMEOUT`
  - 业务类：`INSUFFICIENT_CREDITS`、`MODEL_NOT_ALLOWED`、`EMPTY_PROMPT`、`INVALID_IMAGE_URL`
  - 上游透传：`UPSTREAM_4XX`/`UPSTREAM_5XX` 或按现有后端 code 归一

验收标准：

- UI 不再判断 `error.message.includes(...)`，只根据 `error.code` 做分支
- 埋点/上报永远带 `taskId + error.code + traceId`

### 16.3 History 的“统一视图”策略（guest + authed）

建议把历史定义为 Task 列表的“统一视图”，数据源可多路合并，但 UI 只看一份：

- Data sources：
  - guest：IndexedDB（localForage）或 localStorage（现状可先不动，但建议迁移到 localForage 以支持容量与索引）
  - authed：server history
- 合并策略（建议最小可行）：
  - 登录后弹一次提示：是否迁移 guest 历史到账号（可跳过）
  - 若用户选择迁移：逐条上报 PromptSpec + 缩略图（缩略图可选），成功后清空 guest
  - 若跳过：guest 仍保留，但 UI 切换到 server 为主（可提供“本地历史”过滤）

存储策略建议：

- 本地只存：缩略图（短边 360）+ PromptSpec（v2）+ outputUrls；不存原图二进制
- 容量上限：例如 120 条（按 LRU 淘汰），避免无限增长拖慢启动

### 16.4 Settings 的快照化（contextSnapshot）

当前 `stores/settings.ts` 的产品档案会随用户修改而变化，但历史复现必须固定当时的 context。建议：

- 生成任务创建时：调用 `buildProductProfileContextText()` 得到 `contextSnapshot` 写入 Task
- 后续 UI 展示历史：只渲染 snapshot，不重新计算（避免“历史被产品档案改写”）

### 16.5 i18n 与文案体系对齐（避免 agentImg 变成孤岛）

agentImg 当前 `useLanguageStore + 自己字典` 能跑，但后续跨页复用会痛。建议落地一个“渐进对齐”路径：

- Phase 0/1：agentImg 内部继续用现有 `t(zh,en)`，但统一把文案 key 收敛到一个对象（减少散落字符串）
- Phase 2：把核心用户可见错误文案与下载/扣费/登录提示迁移到全站 `$t` 体系（不要求一次迁完）
- 目标：错误码到文案映射统一在一个地方（domain 或 infra），UI 不拼装文案

## 17. 性能与稳定性补齐（更多可执行项）

### 17.1 生成链路的“端到端耗时分解”落地方式

建议在 Task.telemetry 内区分阶段耗时，并在埋点里上报：

- `prepareMs`：从点击生成到发出首个请求（含图片缩略图/压缩/写草稿历史）
- `analyzeMs`：deep directions 请求耗时
- `generateMs`：img2img 请求耗时
- `totalMs`：从点击到成功/失败/取消

验收方式：

- 线上能看 P50/P95 的 prepareMs，定位是否卡在图片处理/序列化/写缓存

### 17.2 关键稳定性策略：失败分级与可恢复性

错误分级建议（UI 行为差异明确）：

- 用户可修复（不重试）：`EMPTY_PROMPT`、`INVALID_IMAGE_URL`、`MODEL_NOT_ALLOWED`、`INSUFFICIENT_CREDITS`
- 系统可恢复（可重试）：`UPSTREAM_TIMEOUT`、`API_ERROR_502/503/504`、`FETCH_ERROR`
- 需要全站兜底（引导登录/付费）：`needLogin/needPay=true` 的错误

对应 UI 规范：

- 可重试错误：提供“一键重试”并带原 taskId 的新 task（或明确生成 new taskId，但保留 parentTaskId）
- 需要登录/付费：不在 agentImg 自己弹复杂窗，交给全站体系
- 取消：只显示“已停止”，不混淆为失败

### 17.3 内存与资源治理（图片页常见隐患）

可执行清单：

- ObjectURL：创建后在不需要时 revoke（预览/下载）
- base64：避免把原图 base64 长期存进响应式状态；只保留缩略图 dataUrl
- 历史列表：长列表虚拟滚动或分页渲染（至少做到“只渲染可视区附近”）
- 事件监听：所有 window/document listener 在 onUnmounted 清理

验收方式：

- 连续生成 30 次后内存不持续增长（Chrome Performance memory snapshot 对比）

### 17.4 网络与缓存策略（以“移动端弱网”作为基准）

建议项：

- history：server 分页 + cursor，首屏只拉最近 20 条，滚动再加载
- 图片：结果图走 CDN，设置合理 cache-control；缩略图优先加载
- 请求：对 img2img 设置明确超时策略与用户提示（避免“无反馈等待”）

验收方式：

- Slow 3G 下仍能在 3s 内进入可操作状态（哪怕历史后置加载）

### 17.5 Worker 化的渐进落地细节（避免一次性大改）

建议先做最小可行：

- Worker 输入：`File`（通过 `postMessage` 传递，注意结构化克隆成本）
- Worker 输出：`thumbDataUrl` + `uploadBlob`（如需压缩）
- 主线程只做：读取 File 引用、展示 loading、接收结果、更新状态

如果 OffscreenCanvas 不可用：

- Worker 内仍可用 `createImageBitmap` + canvas（兼容性需验证），不可用时再回退主线程方案（但要节流）

## 19. 未来方向（产品形态、能力边界、商业闭环）

本节强调“未来 6–24 个月 agentImg 应该变成什么”，并把方向落到可以持续迭代的能力面，而不是一次性功能堆砌。

### 19.1 产品形态演进：从“出图页”到“商业视觉生产线”

建议把 agentImg 的最终形态定为一个“面向电商与品牌视觉的生产线”，核心是可复用、可管控、可规模化：

- 任务（Task）是一级对象：生成、变体、上采样、导出、抠图、换背景、局部编辑，都围绕 Task 的状态机流转
- 资产（Asset）是二级对象：参考图、Logo、模板、风格包、导出规格，均可跨任务复用
- 模板（Template）是增长杠杆：把一条成熟的 PromptSpec + 参数 + 导出规格沉淀为模板，降低新用户首次成功门槛

### 19.2 核心能力方向（按“可复用”优先级排序）

- 工作流与任务：统一状态机、可重试/可恢复、子流程（analyze/generate/postprocess）可观测
- 资产管理：历史 = Task 列表视图，收藏/标签/搜索/对比，模板化复用
- 多图与对比：多变体网格 + 最佳选择 + 对比视图 + 一键继续迭代（续写/微调）
- 编辑链路：背景替换、抠图、局部修复、文字/瑕疵清除等“后处理任务化”
- 导出交付：按平台规格导出（尺寸、格式、色彩空间、压缩策略、水印策略）并可追溯

### 19.3 商业闭环方向（点数/付费/转化）

- 费用透明：所有会消耗点数的动作都先展示 costEstimate（按 n、模型、后处理 op）
- 可控试用：新用户“低成本高成功率”的引导路径（模板引导、默认模型策略、弱参数自由度）
- 复购触发：收藏模板、批量生成、批量导出、团队共享等提升留存与复购

## 20. 架构升级方向（从业务模块到可插拔工作流平台）

本节描述 agentImg 的“升级路线”：先把边界收敛，再逐步平台化，最终让新能力可以以低风险方式接入。

### 20.1 演进目标：可插拔的 Workflow + Ports 架构

目标结构：

- UI：只关心 task 列表渲染与事件派发
- Workflows：业务编排入口，维护任务状态机，调用 Ports
- Domain：只放类型、策略、纯函数与 schema 迁移
- Ports：把外部能力抽象为接口（auth/credits/history/prompt/imageGen/postprocess/analytics）
- Adapters：把现有 composables/services/stores 适配成 Ports

收益：

- 新能力只需新增/替换某个 workflow 或 port adapter，不需要在 index.vue 里加更多监听器
- 单测可以绕开 UI，直接验证 workflow 的事件与状态转移
- 接入新的后端接口或新的模型供应商，只替换 ImageGenPort

### 20.2 工作流平台化：把“功能”变成“能力位 + 配置”

建议把下列内容逐步配置化（本地配置 → 后端下发 → 灰度开关）：

- 模型策略：哪些模型对哪些用户/套餐可用、默认模型、fallback 模型
- 成本策略：不同模型、不同 imageCount、不同后处理 op 的 cost 计算方式
- 下载策略：分辨率能力位、是否允许无水印、是否需要 postprocess
- prompt 策略版本：baseStyleVersion/safeNegativeVersion 的版本与切换

配套需要：

- Feature Flags：让高风险能力（新模型/新后处理）可以小流量灰度，快速回滚
- 能力位对齐：前端 UI 不推断能力，全部以策略/能力位渲染（避免“UI 假入口”）

### 20.3 任务编排升级：引入“子流程”与“父子任务”

建议在 Task 上增加两个概念（不要求一次性落地）：

- `parentTaskId`：变体/重跑/后处理产生的新任务，与原任务形成链路
- `spans`：analyze/generate/postprocess 子流程（每个子流程都有 status/timing/error）

收益：

- 一次失败不会让整个任务链失去上下文
- 可观测性可以定位到底是“图片处理慢”还是“上游队列慢”

### 20.4 Prompt 体系升级：PromptSpec 变成“可迁移 schema”

未来两类变化会反复发生：

- 模型变化：不同模型对 prompt 的敏感点不同
- 策略变化：安全负向、风格包、品牌约束会迭代

因此 PromptSpec 必须具备：

- schema 版本：v2/v3…
- 迁移函数：读取旧版本历史时可升级到新版本结构（兼容渲染与复现）
- 可重现字段：modelId、params、seeds、refs 摘要（必要时做 hash）

## 21. 开发方向（工程化、质量、交付节奏）

### 21.1 质量体系：从“能跑”到“可持续迭代”

建议把测试目标拆成三层（优先保障核心链路）：

- 纯函数单测：normalize/parse/policy/cost/promptSpec migration
- workflow 单测：状态机转移、取消/替换、重试策略、埋点字段一致性
- 端到端回归：quick/deep、带 ref/不带 ref、多变体、下载/上采样（最小路径即可）

### 21.2 发布与灰度：为“模型与后处理”这种高风险变更做护栏

- 灰度开关：新模型、新 prompt 版本、新后处理 op 必须可灰度、可回滚
- 回滚策略：一键切回旧模型策略与旧 prompt 版本，避免线上成功率断崖
- 版本可追溯：每条任务写入 `policies.*Version`，便于复盘与对账

### 21.3 观测与 SLO：把北极星目标变成工程约束

建议把以下指标变成“持续监控 + 阈值告警”：

- successRate（按 modelId/mode/hasRef 分组）
- P50/P95 totalMs/prepareMs/analyzeMs/generateMs
- cancelRate（按原因 user/timeout/replaced）
- errorCode TopN（7 天滚动）
- 付费漏斗：landing → 首次生成 → 下载 → 购买（按渠道分组）
