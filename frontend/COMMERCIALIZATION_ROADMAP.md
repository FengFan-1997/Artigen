## 📊 1. 现状评估：距离“上线挣钱”缺了什么？

经过对 `backend/imgagent`（后端支付核心）、`backend/server.js`（鉴权与路由）以及 `frontend/src/agentImg/views/AetherMarket.vue`（前端商城）的深度审计，目前的结论是：**“有店面，没收银台”。**

### 🔴 核心阻碍 (Showstoppers) - 必须解决才能上线
1.  **前端“购买”按钮是摆设**：
    *   `AetherMarket.vue` 中的 `<button class="buy-btn">` 没有任何点击事件绑定（`@click` 为空）。用户点击后毫无反应。
    *   **缺失逻辑**：点击按钮 -> 调用创建订单接口 -> 跳转支付页面/展示二维码。
2.  **后端缺主动支付网关**：
    *   后端目前只有 `api/pay/afdian/webhook`（爱发电被动回调）和 `api/credits/order/mock`（测试用模拟接口）。
    *   **缺失集成**：完全没有 Stripe、微信支付、支付宝或类似 PayJS/Xunhu 等聚合支付的主动发起接口。你无法让用户发起一笔支付。
3.  **支付流程闭环断裂**：
    *   用户无法在前端发起订单。
    *   后端无法创建待支付订单并返回支付链接。
    *   没有轮询或回调机制通知前端支付成功。

### 🟡 重要体验缺失 (User Experience Gaps)
1.  **鉴权与转化的冲突**：
    *   后端 `assertAuthedUserMatches` 强制要求所有积分相关接口必须登录。
    *   前端商城页面看起来允许游客浏览，但点击购买时若未登录，没有自动弹窗引导登录的逻辑（因为按钮本身没逻辑）。
2.  **订单历史不可见**：
    *   后端有 `/api/credits/orders` 接口，但前端似乎没有对应的“我的订单”页面供用户核对消费记录。
3.  **缺乏套餐差异化落实**：
    *   前端展示了“Starter”, “Standard”, “Pro”, “Ultimate” 四种包。
    *   后端目前逻辑似乎只有“充值积分” (`credits`)。需要确认购买“Pro包”是否只是增加积分，还是会赋予特殊权限（如 `ui.proFeature2: 解锁 Pro 高清模型`）。如果只是纯积分，这属于文案包装；如果涉及权限，后端 `profiles.js` 需要支持 VIP 字段。

## 🗺️ 2. 未来发展路线图 (Development Roadmap)

### 第一阶段：打通收银台 (MVP - Minimum Viable Product)
**目标**：让用户能真正把钱付进来，积分能自动到账。

*   **后端开发**：
    *   **选型**：决定支付渠道。
        *   *国内个人开发者*：推荐 **爱发电 (Afdian)** (已有一半代码) 或 **PayJS/虎皮椒** (微信/支付宝免签)。
        *   *出海/企业*：必须接 **Stripe**。
    *   **接口实现**：
        *   `POST /api/pay/create-order`: 接收套餐ID，返回支付链接/二维码。
        *   完善 `POST /api/pay/webhook`: 确保第三方回调能准确增加积分并记录日志。
*   **前端开发**：
    *   **商城交互**：给 `buy-btn` 绑定点击事件。
    *   **支付流程**：点击 -> 检查登录 -> (未登录则弹窗) -> 调用创建订单 -> 弹窗显示二维码或跳转 -> 轮询订单状态 -> 支付成功动画 -> 刷新余额。

### 第二阶段：增强留存与复购 (Retention & LTV)
**目标**：让用户买了一次还想买，且觉得物超所值。

*   **功能模块**：
    *   **会员订阅制 (Subscription)**：除了买积分，增加“月卡/年卡”。月卡用户每天送 50 积分，比单买划算。这能极大锁定用户长期留存。
    *   **签到奖励**：`backend/imgagent` 中已有 `/api/credits/checkin` 占位符，需要实现逻辑：每天点一下送 1-5 积分。
    *   **邀请返利 (Referral)**：生成邀请码，邀请朋友注册，双方各得 50 积分。这是低成本获客神器。

### 第三阶段：精细化运营 (Operations & Scale)
**目标**：通过数据分析优化转化率。

*   **数据埋点**：
    *   记录：进入商城人数 vs 点击购买人数 vs 支付成功人数。
    *   分析：哪个套餐卖得最好？用户在哪个步骤流失了？
*   **动态定价**：
    *   后端支持配置热更新，促销节日（双11、黑五）自动打折。

## 🎯 3. 详细执行文档：从零到一的补全计划

为了让你尽快上线挣钱，我为你拆解了**未来 3 天**的紧急开发任务列表：

### 📅 Day 1: 支付核心 (The Cash Register)
1.  **后端**：完善 `imgagent/credits.js`。
    *   确认套餐配置（ID, 价格, 积分数）是硬编码在代码里还是数据库里？建议先搞个 `const PACKAGES = {...}` 常量配置。
    *   实现 `createOrder(userId, packageId)` 方法，生成唯一的 `orderId` 并存入 `orders` 数组（状态: `pending`）。
2.  **后端**：实现简单的支付链接生成。
    *   如果用爱发电，生成的链接是固定的，但需要让用户填备注（你的 orderId）。或者对接其 API 自动生成。
    *   *建议*：为了快速上线，先做**“扫码转账+人工/半自动核销”**或者**“爱发电自动发卡”**模式。

### 📅 Day 2: 前端交互 (The Counter)
1.  **AetherMarket.vue**：
    *   引入 `useUserStore` 获取用户登录状态。
    *   实现 `handleBuy(packageId)` 函数。
    *   逻辑：`if (!user.isLoggedIn) return showLoginModal();`
    *   逻辑：`const { payUrl } = await api.createOrder(packageId); window.open(payUrl);`
2.  **状态同步**：
    *   用户支付完回来，页面要能自动刷新余额。

### 📅 Day 3: 测试与部署 (Verification)
1.  **全链路测试**：
    *   自己注册一个小号，真实走一遍流程。
    *   确保积分扣除（生成图片）和积分增加（充值）的日志都准确无误。
2.  **安全检查**：
    *   确保没人能通过伪造请求直接调用 `addCredits` 接口（目前代码看起来是安全的，因为 `credits.js` 没直接暴露加钱接口给 HTTP，只有 webhook 能加钱）。

### 💡 产品经理的特别建议

> **"不要等到完美才上线。"**

你现在的系统，**画图功能已经很棒了**（基于 SiliconFlow/Gemini）。
现在的瓶颈就是**“无法收费”**。

**我的建议方案（最快上线路径）**：
利用现有的 `api/pay/afdian/webhook`。
1.  **前端**：点击购买 -> 弹窗显示你的爱发电主页链接，并提示“请备注您的用户ID：User_123”。
2.  **爱发电后台**：配置 Webhook URL 指向你的服务器。
3.  **用户**：在爱发电支付 -> 爱发电回调你的服务器 -> 服务器给该 User_123 加分。

这是最不需要写新代码、只需配置的路径。虽然体验稍显原始，但能立马开始验证商业模式！
